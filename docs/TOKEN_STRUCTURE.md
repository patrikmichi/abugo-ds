# Token Structure

This document describes the subfolder structure for design tokens.

## Overview

Design tokens are organized into subfolders for better maintainability and organization. The main token files (`primitives.json`, `semanticTokens.json`, `componentTokens.json`) are generated from these subfolders using build scripts.

## Directory Structure

```
tokens/
├── primitives/           # Base design tokens
│   ├── colors.json
│   ├── typography.json
│   ├── spacing.json
│   ├── radius.json
│   ├── shadows.json
│   ├── border.json
│   ├── sizing.json
│   ├── opacity.json
│   ├── animation.json
│   ├── zIndex.json
│   └── universal.json
├── semanticTokens/       # Semantic design tokens
│   ├── background.json
│   ├── content.json
│   ├── border.json
│   ├── radius.json
│   ├── typography.json
│   ├── icon.json
│   ├── gap.json
│   ├── padding.json
│   ├── control.json
│   ├── shadow.json
│   ├── zIndex.json
│   ├── opacity.json
│   ├── animation.json
│   ├── sizing.json
│   └── borderWidth.json
├── componentTokens/      # Component-specific tokens (hybrid approach)
│   ├── shared/           # Category-first: shared properties across components
│   │   ├── radius.json   # radius.button.sm, radius.field.md, etc.
│   │   ├── gap.json      # gap.button.sm, gap.field.md, etc.
│   │   ├── padding.json  # padding.button.sm, padding.field.md, etc.
│   │   ├── shadow.json   # shadow.card, shadow.modal, etc.
│   │   ├── fontSize.json # fontSize.button.sm, fontSize.field.md, etc.
│   │   ├── lineHeight.json
│   │   ├── animation.json
│   │   ├── borderWidth.json
│   │   ├── zIndex.json
│   │   └── icon.json
│   └── components/       # Component-first: unique component properties
│       ├── button.json   # button.primary.background.default, etc.
│       ├── field.json    # field.height.sm, field.width.full, etc.
│       ├── alert.json
│       └── ... (one file per component)
├── primitives.json      # Generated (merged from primitives/)
├── semanticTokens.json  # Generated (merged from semanticTokens/)
└── componentTokens.json # Generated (merged from componentTokens/)
```

## Working with Tokens

### Editing Tokens

**Always edit tokens in the subfolders**, not in the main JSON files. The main files are auto-generated.

### Building Tokens

To merge subfolders into main JSON files (for Tokens Studio):

```bash
npm run build:tokens
```

### Splitting Tokens

To split main JSON files back into subfolders (if needed):

```bash
npm run split:tokens
```

## Integration

### Tokens Studio (Figma)

Tokens Studio reads from the main JSON files (`primitives.json`, `semanticTokens.json`, `componentTokens.json`). These are generated by running `npm run build:tokens`.

### Storybook

Storybook automatically loads tokens from subfolders using `scripts/load-tokens.ts`. No build step required.

## Component Token Hybrid Approach

Component tokens use a **hybrid approach** that combines category-first organization for shared properties and component-first organization for unique properties.

**See [Component Token Rules](COMPONENT_TOKEN_RULES.md) for complete documentation, decision tree, and examples.**

## Benefits

1. **Better Organization**: Tokens are grouped by category, making them easier to find and maintain
2. **Reduced Merge Conflicts**: Multiple team members can work on different categories simultaneously
3. **Clearer Structure**: Each file has a focused purpose
4. **Easier Navigation**: Smaller files are easier to navigate in IDEs
5. **Flexible Workflow**: Can work with subfolders directly or use merged files for Tokens Studio
6. **Hybrid Approach**: Combines category-first for shared properties and component-first for unique properties
