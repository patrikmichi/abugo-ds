# Commit Changes

Commit staged and unstaged changes to the current branch following project conventions.

## Steps

1. Run `git status` and `git diff` (staged + unstaged) to understand all changes.
2. Run `git log --oneline -5` to see recent commit style.
3. Get the current branch name via `git rev-parse --abbrev-ref HEAD`.

## Safety checks

- **Working branch**: The working branch is `patrik-hackuje`. If not on this branch, STOP and ask the user if they want to switch to it before committing.
- **Protected branches**: If on `master`, `main`, or `develop`, STOP and ask the user for confirmation before committing. Do not proceed without explicit approval.
- **Secrets detection**: Scan diffs for env vars, tokens, API keys, passwords, credentials, or `.env` file contents. If anything suspicious is found, STOP and warn the user. Do not commit.

## Splitting commits

If the changes clearly belong to different logical units (e.g. a new feature + a config change + a docs update), create separate commits for each. Stage files selectively with `git add <path>`.

## Commit message format

### Prefix

Derive the prefix from the branch name:

- Branch starts with `new/` -> prefix `New:`
- Branch starts with `fix/` -> prefix `Fix:`
- Branch starts with `refactor/` -> prefix `Refactor:`
- Branch starts with `chore/` -> prefix `Chore:`
- Branch starts with `docs/` -> prefix `Docs:`

If the branch name does not match any of these, choose the most appropriate prefix based on the nature of the changes:

- `New:` — New feature or functionality added
- `Fix:` — Bug fix or error correction
- `Refactor:` — Code restructuring without changing functionality
- `Chore:` — Maintenance tasks, dependency updates, config changes, adding tests
- `Docs:` — Documentation updates, README changes, code comments

### Suffix

If the branch name ends with a ticket reference pattern (e.g. `-DEV-1234`, `-PROJ-567`), append it to the commit message as `- DEV-1234`.

### Message body

Keep the message short and to the point. No unnecessary details. Describe *what* changed, not *how*.

### Examples

Branch: `new/calendar-page-DEV-1234`
```
New: Add calendar page header components - DEV-1234
```

Branch: `fix/login-redirect-PROJ-42`
```
Fix: Resolve login redirect loop - PROJ-42
```

Branch: `chore/update-deps`
```
Chore: Update dependencies
```

## Execution

After composing the message, stage the relevant files and create the commit(s). Do NOT push to remote unless explicitly asked.
